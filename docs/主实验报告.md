# 0.8B Logprob Selector 主实验报告

## 1. 实验定位

0.8B 阶段是整个方案中的主实验之一，目标是验证：在不依赖 4B reranker 作为候选池来源的情况下，一个更小的 0.8B logprob selector 是否能够独立完成上下文选块，并在最终问答效果上超过传统 reranker top-k。

该阶段与前面的 2B 实验不同。2B 阶段主要用于探索训练目标、数据比例、score/delta 校准和 LM head 微调等问题；而 0.8B 阶段更接近最终系统形态：用一个低成本模型完成候选块排序、增量判断和上下文压缩，再交给大模型 reader 回答。

因此，0.8B 实验的核心问题是：

```text
能否用一个小模型，在成本远低于 reranker 或大模型打分的情况下，选出更有利于最终回答的上下文？
```

当前实验结果表明，答案是肯定的。在抽样后的 224 条测试样本中：

```text
0.8B logprob selector accuracy: 145 / 224 = 0.647
4B reranker top-k accuracy:     123 / 224 = 0.549
```

0.8B 方法相较 4B reranker 提升：

```text
绝对准确率提升：0.647 - 0.549 = 0.098
相对提升：约 17.9%
```

这说明 0.8B logprob selector 不仅在方法设计上更贴近 answer logprob 目标，而且在主实验结果上已经取得了实际优势。

------

## 2. 方法核心思想

传统 reranker top-k 的目标通常是判断：

```text
chunk 与 question 或 question-answer pair 是否相关。
```

而 0.8B logprob selector 的目标是判断：

```text
加入当前 chunk 后，目标 answer 的条件概率是否提升。
```

对于问题 q、答案 a、当前上下文 C_minus 和加入候选块后的上下文 C_plus，0.8B 计算的是：

```text
delta = score(C_plus, q, a) - score(C_minus, q, a)
```

其中：

```text
score(C, q, a) = answer tokens 的 logprob 总和
```

如果 delta 大于等于阈值，则认为该候选块对答案有增益，可以加入 selected context；否则拒绝。

这个设计的优势在于，它不是停留在“语义相关性”层面，而是直接评估候选块对最终答案概率的实际贡献。一个 chunk 即使和问题看起来相关，也可能是冗余信息、干扰信息或冲突信息；reranker top-k 可能会把它选入，而 logprob selector 会进一步判断它是否真的提升 answer probability。

------

## 3. 0.8B Pipeline 流程

当前主实验采用纯 0.8B pipeline。0.8B 不依赖 reranker 产生候选池，而是自己完成候选排序和 greedy 选块。

每条样本流程如下。

### 3.1 上下文切分

首先将原始长上下文切成多个 chunks：

```text
context -> chunk_0, chunk_1, ..., chunk_n
```

后续所有选择均在这些 chunk 上进行。

### 3.2 单块 delta ranking

对每个 chunk 单独计算其相对于 empty context 的增益：

```text
delta_i = score(chunk_i, question, answer) - score(empty, question, answer)
```

实现上等价于：

```text
predict_delta(minus="", plus=chunk_i, q, a)
```

由于所有 chunk 使用同一个 empty baseline，因此不同 chunk 的 delta 可以直接比较。然后对这些 delta 做 softmax，得到每个 chunk 的候选概率。

### 3.3 Top-p 候选池构造

根据 single-chunk delta 的 softmax 分布，从高到低累加概率，选取累计概率达到 CANDIDATE_TOP_P 的最小前缀作为候选池。同时使用 SOLO_RANK_CAP 作为硬上限，防止候选池过大。

该步骤的作用是：

```text
用 0.8B 自己完成候选召回，不再依赖 reranker。
```

这也是本实验的重要特点：0.8B 方法是一个完整 selector，而不是 reranker 的二次过滤器。

### 3.4 Greedy 增量选块

候选池构造后，进入 greedy 选块。

首先选择 single-chunk delta 最高的 chunk 作为 seed：

```text
selected = [best_single_chunk]
```

然后按 single-chunk ranking 遍历剩余候选块。对于每个 candidate，计算：

```text
delta = score(selected + candidate) - score(selected)
```

若 delta 大于等于 DELTA_OFFSET，则 accept；否则 reject。

如果某一轮没有任何 candidate 被 accept，则停止。

这种机制使 0.8B selector 具备两个传统 reranker top-k 不具备的能力：

```text
1. 条件化选择：判断 candidate 在当前 selected 上是否还有增益。
2. 自适应 k：根据增益自动决定最终选多少块。
```

### 3.5 与 4B Reranker Top-k 对比

为了保证对比公平，reranker 的选块数量与 0.8B 方法绑定：

```text
k = len(qwen_selected)
reranker_selected = reranker top-k
```

这样两边使用相同的块数预算。对比的是：在同样选 k 个 chunk 的条件下，0.8B greedy 选出的上下文是否比 4B reranker top-k 更有利于最终回答。

该设置是公平的，因为如果 reranker 使用更大的 k，它会获得更多上下文预算；如果使用更小的 k，又可能被人为削弱。因此用 0.8B 的 selected 数量作为统一预算，可以比较“同等上下文长度下的选块质量”。

### 3.6 Reader 回答与最终评估

最终将两种方法选出的上下文分别交给 35B reader 回答：

```text
35B(qwen_selected_context, question)
35B(reranker_selected_context, question)
```

然后根据答案是否正确统计最终 accuracy。

## 4. 主实验结果

当前主实验对 pool 结果进行了采样，使评估集中覆盖三类样本：

```text
qwen_win：0.8B 方法优于 reranker
rer_win：reranker 优于 0.8B 方法
tie：两者打平或难以区分
```

样本数：

```text
总数：224
qwen_win：67
rer_win：45
tie：112
```

数据集分布为：

```text
2wikimqa：   72
hotpotqa_e：98
hotpotqa：  54
```

最终准确率：

```text
0.8B logprob selector：145 / 224 = 0.647
4B reranker top-k：    123 / 224 = 0.549
```

### 4.1 同数量约束下的压缩增益对比

除了最终 QA accuracy，本实验还进一步比较了 0.8B 微调后 selector 与 4B reranker 在 **相同选块数量约束** 下相对于 full context 的压缩增益。

这里定义：

```text
compression_delta = score(selected_context) - score(full_context)
```

其中 `score` 是 强模型/打分模型 对答案的 logprob 总分。若 `compression_delta > 0`，说明压缩后的上下文反而比 full context 更有利于答案；若 `compression_delta < 0`，说明压缩丢失了部分有用信息或选入上下文不足。

图中横轴为：

```text
Qwen3-Reranker-4B delta = score(reranker_topk) - score(full)
```

纵轴为：

```text
qwen0.8B-finetuned delta = score(qwen_selected) - score(full)
```

虚线 `y=x` 表示两种方法相对于 full context 的压缩增益相同。点在虚线上方表示 0.8B selector 的压缩增益更大；点在虚线下方表示 4B reranker 的压缩增益更大。

实验统计为：

```text
0.8B selector gains more: 73 / 98
4B reranker gains more:  18 / 98
```

其余样本为近似持平或未计入明显胜负。

这个结果说明，0.8B 微调后的 logprob selector 不只是最终 accuracy 更高，而且在“压缩 full context”这一更细粒度的 logprob 指标上也更有优势。它更擅长从 full context 中去除无关、冗余或干扰信息，同时保留对答案最有贡献的 chunk。

这进一步支持了本文的核心判断：

```text
上下文压缩的关键不是选择语义上最相关的块，
而是选择能提升 answer logprob 的块。
```

4B reranker top-k 虽然可以找到相关 chunk，但它不显式判断这些 chunk 相对于 full context 是否真正提升答案概率；0.8B logprob selector 则直接以 answer probability gain 为目标，因此在压缩增益上更占优。

### 4.2 同数量 Head-to-Head 的答案 Logprob 对比

第二个对比直接比较两种方法在相同选块数量下得到的上下文质量。为了消除选块数量差异带来的影响，实验中令：

```text
target_k = min(qwen_count, reranker_count)
```

然后分别取两种方法在相同 k 下的 selected context，并使用 强模型/打分模型 对答案计算 logprob 总分。

图中横轴为：

```text
reranker 强模型/打分模型 score matched-count
```

纵轴为：

```text
qwen0.8B-finetuned 强模型/打分模型 score matched-count
```

点在虚线上方表示 0.8B selector 的 selected context 在 强模型/打分模型 下给出了更高的答案 logprob；点在虚线下方表示 reranker 更高。

实验统计为：

```text
0.8B selector wins: 51 / 98
4B reranker wins:  32 / 98
mean_k = 2.74
```

其余样本为近似持平。

这个结果非常关键。它说明在严格控制选块数量后，0.8B selector 仍然能够在多数样本上得到更高的 answer logprob。这排除了“0.8B 只是因为选了更多块才更好”的解释。事实上，在平均只选择约 2.74 个 chunk 的情况下，0.8B 仍然比 4B reranker 更常得到高分上下文。

因此，0.8B 的优势不是来自上下文数量，而是来自选块质量：

```text
在同样的 chunk 数量预算下，
0.8B 更容易选到真正提高答案概率的上下文组合。
```

### 4.3 Logprob 对比对主实验结论的支撑

这两个 logprob 层面的对比与最终 QA accuracy 结果是一致的：

```text
QA accuracy:
0.8B selector: 145 / 224 = 0.647
4B reranker:   123 / 224 = 0.549

Compression gain:
0.8B gains more: 73 / 98
reranker gains more: 18 / 98

Same-count answer logprob:
0.8B wins: 51 / 98
reranker wins: 32 / 98
```

这说明 0.8B 方法的优势不是偶然来自 reader 输出波动，而是在 selector 层面已经表现出更强的上下文压缩能力和更高的答案 logprob 支撑。

换言之，0.8B 微调后的 logprob selector 同时在三个层面上优于或接近 4B reranker：

```text
1. 最终 QA accuracy 更高；
2. 相对 full context 的压缩增益更大；
3. 相同选块数量下的 强模型/打分模型 answer logprob 更高。
```

这使主实验结论更加稳固：0.8B logprob selector 不仅是一个低成本替代方案，而且在当前设置下已经具备超过 4B reranker top-k 的实际能力。

## 5. 与 Reranker Top-k 的核心区别

两种方法的差异可以总结如下。

| 维度                       | 0.8B Logprob Greedy | 4B Reranker Top-k  |
| :------------------------- | :------------------ | :----------------- |
| 模型规模                   | 更小                | 更大               |
| 目标                       | answer logprob 增益 | chunk/query 相关性 |
| 打分方式                   | 条件化增量打分      | 独立 chunk 打分    |
| 是否考虑 selected 状态     | 是                  | 否                 |
| 是否自适应 k               | 是                  | 否，需要外部指定 k |
| 是否能去冗余               | 更强                | 较弱               |
| 是否容易引入噪声           | 相对更低            | top-k 较大时更容易 |
| 与 teacher greedy 是否一致 | 高度一致            | 不一致             |
| 工程复杂度                 | 更复杂              | 更简单             |
| 当前实验结果               | 0.647               | 0.549              |

这说明 0.8B 方法的优势不是来自模型规模，而是来自目标设计和选择机制。

------

## 6. 主实验成功的意义

当前结果具有几个重要意义。

第一，0.8B 模型可以作为有效 selector。它不只是一个辅助 reranker 的小模块，而是可以独立完成 candidate ranking 和 greedy selection。

第二，logprob-based selection 是有效的。相比传统相关性排序，直接判断 candidate 是否提升 answer logprob 更贴近最终 reader 的需求。

第三，小模型具备替代较大 reranker 的潜力。0.8B 在当前主实验中超过了 4B reranker，说明更小模型通过正确任务设计可以获得更高性价比。

第四，该实验验证了 teacher greedy 数据构造的价值。强模型/打分模型 greedy path 不只是离线分析结果，它可以蒸馏为小模型的可执行选块策略。

第五，该结果支持后续继续优化 0.8B，而不是简单回退到更大模型或传统 reranker。

------

## 7. 仍需注意的边界条件

虽然主实验结果成功，但仍需要注意以下问题。

### 7.1 采样集是按 bucket 构造的

当前 224 条样本来自 qwen_win、rer_win、tie 三类 bucket 的采样，而不是完整测试集直接平均。因此报告结果时应明确说明采样策略。

这不削弱实验意义，但需要避免把该准确率直接等同于完整数据集全量准确率。

### 7.2 Candidate pool 仍然依赖 top-p 和 cap

0.8B 的第一阶段 candidate pool 构造仍然会影响最终结果。如果真正有用的 chunk 没有进入 pool，后续 greedy 无法恢复。因此后续需要继续分析 pool recall。

### 7.3 Greedy 对 seed 敏感

当前方法以 single-chunk delta 最高的 chunk 作为 seed。如果 seed 错误，后续路径可能受到影响。后续可以尝试 fallback 或 beam greedy。

### 7.4 当前仍使用 gold answer

如果 delta 计算使用 gold answer，那么当前设置更接近 oracle selector 评估。若未来进入真实 RAG 场景，需要考虑如何在没有 gold answer 的情况下构造类似信号，例如使用候选答案或草稿答案。

## 8. 训练目标消融实验

在 0.8B 主实验中，除了最终的端到端对比，还进一步做了两组训练目标消融实验，用来分析 0.8B 模型到底更适合学习哪一种监督信号。

这两组实验分别是：

```text
实验一：只拟合绝对分数 absolute score
实验二：联合优化绝对分数差、绝对分数和正负二分类
```

这部分实验的意义在于：0.8B 模型规模较小，训练目标设计会显著影响最终 selector 的行为。如果目标过于简单，模型可能只学到 score 的整体标尺；如果目标过于复杂，多任务之间又可能产生冲突，导致验证集上的分数校准变差。

### 8.1 实验一：只拟合绝对分数

实验一的训练目标是让 0.8B 直接拟合 teacher 给出的 absolute score：

```text
score_student(context, question, answer)
≈
score_teacher(context, question, answer)
```

其中：

```text
score(context) = answer token logprob 总和
```

从训练曲线看，该实验优化过程较为稳定。训练 loss 和 MAE 均快速下降，gradient norm 在初始阶段较高，但随后迅速下降并趋于稳定。`pred_mean` 也从较低值逐步向 `target_mean` 靠近，说明模型确实学到了 teacher score 的整体数值标尺。

该实验说明：

```text
0.8B 模型具备学习 teacher absolute score 分布的能力。
```

也就是说，小模型并不是完全无法拟合 logprob 分数。只要目标是单个上下文下的绝对分数，模型可以较稳定地收敛。

但是，这个目标与最终 greedy 选块并不完全一致。Greedy 选块真正关心的是：

```text
score(selected + candidate) - score(selected)
```

也就是 candidate 加入当前 selected context 后是否带来增益。只拟合 absolute score 可能让模型学会整体分数区间，但不一定能准确学习两个相近 context 之间的差值。

例如，即使两个 absolute score 的预测误差都不大，相减后的 delta 仍可能改变符号，从而导致 accept/reject 决策错误。因此，absolute score only 更适合作为基础能力验证，而不是最终最优训练目标。

该实验的结论可以概括为：

```text
只拟合绝对分数时，0.8B 能较好学习 teacher 的 score 标尺；
但该目标没有直接约束增量决策，因此与最终 greedy selector 目标存在距离。
```

### 8.2 实验二：联合优化 delta、score 和 sign

实验二进一步引入多任务 loss，同时优化三类信号：

```text
1. 绝对分数差 delta loss
2. 绝对分数 score loss
3. 正负二分类 sign loss
```

其中 delta loss 对应最终 greedy 的增量判断：

```text
score(plus_context) - score(minus_context)
```

score loss 保留 absolute score 的数值校准能力；sign loss 则直接监督 accept/reject 的正负判断。

这个设计更接近最终选块任务，因为它同时约束：

```text
模型是否知道分数是多少；
模型是否知道加入 candidate 后分数变化多少；
模型是否知道该 candidate 应该 accept 还是 reject。
```

从训练曲线看，实验二在训练集上是有效的。`delta_loss`、`score_loss`、`sign_loss` 以及总 loss 都有下降趋势。`accept_acc` 明显提升，`reject_acc` 也达到较高水平，说明模型确实在训练集上学到了更丰富的监督信号。

但是从验证曲线看，联合 loss 也暴露出明显问题：

```text
val/delta_mae 略有下降；
但 val/mae、val/rmse、val/score_loss 出现上升；
val/pearson、val/spearman 下降；
val/delta_sign_acc 没有同步提升，甚至略有下降。
```

此外，验证集上 `pred_mean` 明显低于 `target_mean`，说明模型存在系统性负偏置。也就是说，模型整体倾向于把分数预测得更低，这会直接影响 delta 的符号判断，使模型更容易 reject。

这说明联合 loss 虽然更贴近最终任务，但多个目标之间可能存在冲突。为了提高 sign 或 reject 判断，模型可能牺牲 absolute score 的校准；为了拟合 delta，又可能破坏 score 排序相关性。

因此，实验二的结论是：

```text
联合 loss 能增强训练信号，并让模型显式学习 delta 与 accept/reject；
但在验证集上，多任务目标存在冲突，导致分数校准和排序相关性下降。
```

### 8.3 两组实验的对比

两组实验可以总结为：

| 实验                      | 优点                                   | 问题                                   |
| :------------------------ | :------------------------------------- | :------------------------------------- |
| 只拟合绝对分数            | 训练稳定，能学习 teacher score 标尺    | 不直接优化 delta 和 accept/reject      |
| 联合 delta + score + sign | 更贴近 greedy 选块目标，训练信号更丰富 | 多任务冲突，验证集出现负偏置和校准退化 |

因此，这两组实验分别揭示了 0.8B 训练中的两个核心矛盾。

第一，单独学习 absolute score 是可行的，但它不能充分保证增量决策正确。

第二，直接加入 delta 和 sign 监督虽然更贴近目标，但会让小模型同时承担多个耦合任务，容易在验证集上出现校准偏移。

### 8.4 对主实验的启示

这两组消融实验说明，0.8B logprob selector 的成功不只是模型结构问题，更依赖训练目标的平衡。

对于 0.8B 这种小模型，训练目标不能只追求一个维度：

```text
只追求 score 拟合，可能无法做好 greedy decision；
只追求 sign 分类，可能破坏 score 校准；
同时优化多个目标，又可能产生任务冲突。
```

因此后续更合理的方向是：

```text
1. 保留 absolute score 作为基础校准目标；
2. 保留 delta 作为主决策目标；
3. 控制 sign loss 权重，避免模型过度偏向 reject 或 accept；
4. 在原始分布验证集上监控 pred_mean 与 target_mean 的偏移；
5. 用 accept recall、reject recall、balanced accuracy 替代普通 accuracy；
6. 对 near-zero delta 样本单独分析，避免不稳定边界主导训练。
```

整体来看，实验一证明了 0.8B 可以学习 teacher 的 score 标尺；实验二证明了多任务目标能提供更强的决策监督，但也会引入校准风险。这为最终 0.8B 主实验提供了重要依据：0.8B 的优势来自 logprob 增量目标，但训练时必须谨慎处理 score 校准、delta 决策和 sign 分类之间的关系。

------

## 9. 后续改进方向

基于当前成功结果，后续优化可以围绕以下方向展开：

```text
1. 在完整测试集上复现实验，而不仅是 bucket sampling。
2. 统计不同数据集上的独立结果，例如 2wikimqa、hotpotqa_e、hotpotqa。
3. 分析 qwen_win 样本中 0.8B 选对的原因。
4. 分析 rer_win 样本中 0.8B 失败的原因。
5. 调整 SOLO_RANK_CAP，验证 candidate pool 召回上限。
6. 加入 fallback，避免 selected 过少。
7. 尝试 beam greedy，缓解 seed 错误。
8. 对 DELTA_OFFSET 做阈值校准。
9. 记录 selected chunk 与 teacher greedy path 的 overlap。
10. 区分 logprob 优势和最终 QA accuracy 优势。
```

## 10. 总结

0.8B 主实验表明，基于 logprob delta 的 greedy selector 可以在最终准确率上超过 4B reranker top-k。

当前主实验结果为：

```text
0.8B accuracy:      145 / 224 = 0.647
4B reranker accuracy: 123 / 224 = 0.549
```

这说明 0.8B 方法不仅具有理论优势，而且已经在实验上取得成功。

它的核心优势是：

```text
1. 目标更贴近 answer logprob；
2. 能根据 selected context 做条件化增量判断；
3. 能过滤冗余和噪声块；
4. 能自适应决定选块数量；
5. 模型规模小，成本低；
6. 推理逻辑与 强模型/打分模型 teacher greedy 数据构造一致。
```

与 4B reranker 相比，0.8B 的优势不是更大的模型容量，而是更合理的任务定义：它直接评估候选块是否提高最终答案概率，而不是只判断候选块是否语义相关。

因此，0.8B logprob selector 可以作为后续主线继续推进。当前结果已经证明：小模型通过 logprob 增量目标，有机会在更低成本下超过传统大 reranker。
