# 基于 Logprob 增量打分的 Greedy 选块数据构造

## 摘要

本文整理了一套用于训练小模型学习上下文选块能力的数据构造方法。该方法使用 teacher model （使用支持logprob的大模型api调用）对目标答案的 token-level logprob 进行打分，并将候选上下文块加入前后的分数变化构造成监督信号。

与传统的 reranker top-k 方案不同，该方法并不只判断某个 chunk 与 question 是否语义相关，而是关注一个更直接的问题：

```text
将当前 candidate chunk 加入已有上下文后，模型对目标答案的条件概率是否提升？
```

因此，数据构造的核心不是简单地给每个 chunk 打一个相关性标签，而是通过逐步 greedy 选块形成一条上下文选择路径，并将路径中的每一步转化为 `minus_context`、`plus_context` 和 `target_delta` 形式的训练样本。

------

## 1. 研究背景与问题定义

在长上下文问答任务中，输入文档通常会被切分为多个 chunk。直接将全部 chunk 输入大模型虽然可以保留完整信息，但也会带来两个问题：

1. 输入长度较长，推理成本较高；
2. 无关、冗余或冲突信息可能干扰模型，使模型对正确答案的置信度下降。

因此，需要从大量候选 chunk 中选择一个更小、更有效的上下文集合。

传统方法通常使用 reranker 对每个 chunk 和 question 的相关性进行排序，然后选取 top-k chunk 作为最终上下文。这类方法能够找到与问题表面相关的文本，但它不一定能判断这些文本加入当前上下文后是否真的有助于生成目标答案。

一个 chunk 可能与问题看起来相关，但它可能只是重复信息，也可能引入歧义或干扰。因此，仅依赖相关性排序并不能完全对应最终问答效果。

本文采用的思路是：使用 teacher model 的 answer logprob 作为上下文质量信号，让数据构造过程直接围绕“是否提升目标答案概率”展开。

------

## 2. 分数定义

对于一条样本，设问题为q，目标答案为$ a=(a_1,a_2,…,a_T)$，上下文为 C。teacher model 对答案a部分计算 token-level logprob 总和：
$$
score(C,q,a)= \sum_{t = 1} ^ Tlog⁡p(at∣C,q,a_{<t})
$$
该分数表示：在给定上下文 C和问题 q的条件下，模型对目标答案 a的支持程度。

由于 logprob 通常为负数，所以数值越大表示越好。例如：

```text
-5 优于 -10
-10 优于 -20
```

在本文的数据构造中，score 不是用来直接回答问题，而是用来衡量不同上下文组合对目标答案的支持程度。

------

## 3. 原始数据结构

每条原始样本包含三个基本部分：

```json
{
  "question": "...",
  "answer": "...",
  "chunks": [
    "chunk 0 text",
    "chunk 1 text",
    "..."
  ]
}
```

其中：

- `question`：当前问答任务的问题；
- `answer`：标准答案或目标答案；
- `chunks`：由原始长上下文切分得到的候选文本块。

后续的选块过程都基于 `chunks` 进行。训练目标不是让小模型直接生成答案，而是让小模型学习：

```text
给定当前 selected context 和一个 candidate chunk，判断加入该 candidate 后是否会提升 answer logprob。
```

------

## 4. 数据构造流程

整体数据构造分为三个阶段：

```text
1. 单块打分与候选排序
2. Greedy 选块并记录路径
3. Replay 路径生成 delta 训练样本
```

### 4.1 Step 1：单块打分与候选排序

首先，对每个 chunk 单独构造输入：

```text
chunk_i + question + answer
```

然后使用 teacher model 计算 answer 部分的 logprob 总和：
$$
s_i=score(c_i,q,a)
$$
得到所有单块分数后，将 chunk 按分数从高到低排序。这个排序并不是最终选块结果，而是用于得到一个初始候选顺序。

该步骤的作用是：

```text
先找出单独看起来较有价值的 chunk，作为后续 greedy 搜索的候选基础。
```

在实际实现中，可以根据单块分数做候选过滤，只保留排名较高或概率质量较集中的一部分 chunk。具体过滤阈值属于实现细节，开源文档中不展开。

### 4.2 Step 2：Greedy 选块并生成路径

候选 chunk 排序完成后，进入 greedy 选块阶段。

首先选择单块分数最高的 chunk 作为 seed：
$$
selected= \left\{ c_{seed} \right\}
$$


并计算当前上下文分数：
$$
current\_score=score(selected,q,a)
$$
之后按照候选排序遍历尚未被接受的 chunk。对于当前候选块 $c_j$，构造加入后的上下文：
$$
C ^ +=selected∪\{c_j\}
$$
并计算：
$$
s^+=score(C^+,q,a)
$$
如果加入 candidate 后分数提升：
$$
s^+>current\_score
$$
则接受该 candidate：
$$
selected←selected∪\{c_j\} \\
current\_score ← s^+
$$
否则拒绝该 candidate，`selected` 和 `current_score` 保持不变。

这个过程会持续进行，直到没有新的 candidate 能够带来分数提升，或者达到预设的停止条件。

### 4.3 Greedy Path 记录格式

每条样本最终会得到一条 `greedy_path`。该路径记录 seed、accept 和 reject 节点。

示例格式如下：

```json
[
  {
    "chunk": 0,
    "action": "seed",
    "score": "...",
    "iteration": 0
  },
  {
    "chunk": 1,
    "action": "reject",
    "score": "...",
    "iteration": 1
  },
  {
    "chunk": 2,
    "action": "accept",
    "score": "...",
    "iteration": 1
  }
]
```

字段含义如下：

- `chunk`：当前被测试或初始化的 chunk 下标；
- `action = seed`：初始化 selected context；
- `action = accept`：加入该 chunk 后 score 提升，因此被接受；
- `action = reject`：加入该 chunk 后 score 没有提升，因此被拒绝；
- `score`：当前候选上下文组合下的 answer logprob 分数；
- `iteration`：greedy 搜索轮次。

`seed` 节点用于初始化路径，不直接生成训练样本。真正用于训练的是后续的 accept / reject 节点。

------

## 5. Delta Score 训练数据构造

### 5.1 构造原则

得到 `greedy_path` 后，需要将它转换为小模型可学习的监督数据。

转换方式是 replay 整条路径，并在 replay 过程中维护当前已接受的上下文集合：
$$
selected\_so\_far
$$


对于每个非 seed 节点，设当前候选 chunk 为$c_j$，构造两个上下文：
$$
C^−=selected\_so\_far
$$

$$
C^+=selected\_so\_far∪\{c_j\}
$$



其中：

- $C^−$：加入 candidate 之前的上下文；
- $C^+$：加入 candidate 之后的上下文。

目标 $\Delta$ 定义为：
$$
\Delta=score(C^+,q,a) − score(C^−,q,a)
$$
在字段上对应为：

```text
target_delta = candidate_score - current_score
```

如果 `target_delta > 0`，说明 candidate 对答案概率有正向贡献；如果 `target_delta <= 0`，说明 candidate 没有带来提升，或者引入了干扰。

### 5.2 Replay 规则

Replay 时需要注意 accept 和 reject 的状态更新不同：

```text
如果当前节点是 accept：
    生成一条训练样本
    将 candidate 加入 selected_so_far
    更新 current_score

如果当前节点是 reject：
    生成一条训练样本
    selected_so_far 不变
    current_score 不变
```

也就是说，accept 节点会改变后续上下文状态，而 reject 节点只生成监督样本，不改变 selected context。

### 5.3 输出样本格式

每个非 seed 节点生成一条训练样本。简化后的样本格式如下：

```json
{
  "question": "...",
  "answer": "...",
  "minus_context": "selected_so_far text",
  "plus_context": "selected_so_far + candidate text",
  "minus_context_ids": [0],
  "plus_context_ids": [0, 2],
  "target_delta": "...",
  "action": "accept"
}
```

其中：

- `minus_context`：加入 candidate 前的上下文；
- `plus_context`：加入 candidate 后的上下文；
- `minus_context_ids`：`minus_context` 对应的 chunk 下标；
- `plus_context_ids`：`plus_context` 对应的 chunk 下标；
- `target_delta`：加入 candidate 后的 score 变化；
- `action`：由 `target_delta` 的符号决定，可作为辅助分析字段。

训练时主要使用连续值 `target_delta`，而不是只使用 `action` 这个二分类标签。

### 5.4 上下文拼接顺序

在构造 `minus_context` 和 `plus_context` 时，需要保证上下文拼接顺序稳定。通常可以按照原始 chunk 下标排序后拼接：

```python
"\n\n".join(chunks[i] for i in sorted(indices))
```

这样做的原因是：虽然 greedy 过程中的接受顺序可能与原文顺序不同，但最终输入给模型的上下文最好保持原始文档顺序，以减少由拼接顺序带来的额外干扰。

------

## 6. 与传统 Reranker Top-k 的区别

传统 reranker top-k 通常对每个 chunk 独立打分，然后选取得分最高的若干 chunk。这个过程主要关注：

```text
chunk 与 question 是否相关
```

而 logprob delta greedy 关注的是：

```text
candidate chunk 在当前 selected context 的基础上，是否还能继续提升 answer logprob
```

两者的区别主要体现在以下几个方面：

| 维度               | Reranker Top-k | Logprob Delta Greedy  |
| :----------------- | :------------- | :-------------------- |
| 打分目标           | 语义相关性     | 答案概率增益          |
| 打分方式           | 单块独立打分   | 条件化增量打分        |
| 是否考虑已有上下文 | 通常不考虑     | 考虑 selected context |
| 是否容易选入冗余块 | 相对更容易     | 可通过 delta 进行过滤 |
| 选块数量           | 通常需要预设 k | 可由增益过程决定      |

因此，logprob delta greedy 的重点不是替代所有 reranker，而是提供一种更贴近最终答案概率的监督数据构造方式。

------

## 7. 方法特点

该数据构造方式具有几个特点。

第一，监督信号直接来自 answer logprob。它不只判断文本是否相关，而是判断文本是否提升目标答案概率。

第二，训练样本是成对构造的。每条样本都包含 `minus_context` 和 `plus_context`，因此模型学习的是加入 candidate 前后的变化，而不是孤立上下文的静态分数。

第三，reject 样本同样有训练价值。一个被拒绝的 chunk 并不只是“无关文本”，而是经过 teacher scoring 后确认没有带来增益的候选块。这类负样本可以帮助小模型学习如何避免加入冗余或干扰信息。

第四，该方法可以将 teacher model 的选块行为蒸馏到 student model 中。teacher scoring 只在数据构造阶段使用，训练完成后，小模型可以在推理阶段近似执行增量选块判断。

------

## 8. 简明流程图

```text
原始样本：question, answer, chunks
        │
        ▼
Teacher 对每个单块计算 answer logprob
        │
        ▼
根据单块分数得到候选 chunk 顺序
        │
        ▼
选择最高分 chunk 作为 seed
        │
        ▼
逐轮遍历 candidate chunk：
  计算 score(selected + candidate)
  若分数提升则 accept
  否则 reject
        │
        ▼
得到 greedy_path
        │
        ▼
Replay greedy_path：
  对每个非 seed 节点构造
  minus_context = selected_so_far
  plus_context = selected_so_far + candidate
  target_delta = score(plus) - score(minus)
        │
        ▼
训练小模型：
  student_delta = score_student(plus) - score_student(minus)
  loss = Huber(student_delta - target_delta)
```
